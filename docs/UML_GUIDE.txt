# UML Diagram Usage Guide

## Quick Reference - How to Convert Mermaid Diagrams to Images

### Method 1: Mermaid Live Editor (Easiest)
1. Go to https://mermaid.live/
2. Copy the diagram code from the UML_DIAGRAMS.md file
3. Paste it into the editor
4. Click "Copy as SVG" or "Download as PNG"
5. Use the image in your documentation

**Advantages:**
- No installation required
- Instant preview
- Multiple export formats
- Easy to edit and regenerate

---

### Method 2: Command Line with Mermaid CLI
```bash
# Install Mermaid CLI globally
npm install -g @mermaid-js/mermaid-cli

# Convert Mermaid file to PNG
mmdc -i diagram.mmd -o diagram.png

# Convert to SVG
mmdc -i diagram.mmd -o diagram.svg -t dark
```

**Requirements:**
- Node.js installed
- Graphviz (for some diagram types)

---

### Method 3: VS Code Integration
1. Install extension: "Markdown Preview Mermaid Support"
2. Open UML_DIAGRAMS.md in VS Code
3. Right-click on any diagram and select "Export as SVG/PNG"
4. Choose save location

**Recommended VS Code Extensions:**
- Markdown Preview Mermaid Support
- Markdown All in One
- Draw.io Integration (alternative)

---

### Method 4: Online Converters
Popular services that support Mermaid:
- **Mermaid.live**: https://mermaid.live/
- **Kroki**: https://kroki.io/
- **PlantUML online editor**: http://www.plantuml.com/plantuml/uml/

## Diagram Descriptions for Documentation

### 1️⃣ Class Diagram
**Purpose:** Shows object-oriented design and class relationships
**Use in:**
- Architecture documentation
- Developer reference
- API design documentation
**Best for:** Technical developers and architects

---

### 2️⃣ Use Case Diagram  
**Purpose:** Shows what users can do in the system
**Use in:**
- Project requirements
- Business logic documentation
- Stakeholder presentations
**Best for:** Project managers, stakeholders, business analysts

---

### 3️⃣ Sequence Diagram
**Purpose:** Shows step-by-step process flow
**Use in:**
- Process documentation
- API workflow explanation
- Error handling flows
**Best for:** New developers, technical writers

---

### 4️⃣ System Architecture Diagram
**Purpose:** Shows how different components interact
**Use in:**
- High-level technical documentation
- Deployment documentation
- System design documentation
**Best for:** DevOps, system architects, new team members

---

### 5️⃣ Entity Relationship Diagram (ERD)
**Purpose:** Shows database schema and relationships
**Use in:**
- Database documentation
- Data model explanation
- SQL script generation guide
**Best for:** Database administrators, backend developers

---

### 6️⃣ Data Flow Diagram
**Purpose:** Shows how data moves through the system
**Use in:**
- Security documentation
- Performance analysis
- Integration documentation
**Best for:** Architects, security reviewers

## Integration with Different Documentation Formats

### Word Document
```
1. Export diagrams as PNG from Mermaid Live
2. Create a Word document
3. Insert images at appropriate sections
4. Add captions and descriptions below each image
5. Add table of contents linking to diagrams
6. Save as .docx file
```

### PDF Document
```
1. Use Mermaid Live to export as SVG
2. Use online converter to convert SVG to PDF
3. Or: Insert images in Word → Export to PDF
4. Include bookmark links to diagrams
```

### Markdown Files
```
The diagrams are already in Markdown format!
- Keep the UML_DIAGRAMS.md file in your repo
- Reference it in your main README
- GitHub automatically renders Mermaid diagrams
```

### Confluence/Wiki
```
1. Export diagrams as PNG
2. Upload to Confluence
3. Or paste Mermaid code directly (if Confluence supports it)
4. Add descriptions as page content
```

### Google Docs
```
1. Export diagrams as PNG from Mermaid Live
2. Insert image in Google Doc
3. Add drawing annotations if needed
4. Share with team
```

## Templates for Documentation Sections

### For Architecture Document
```markdown
## System Architecture

[Include System Architecture Diagram]

### Components
- **Frontend Layer**: HTML pages and JavaScript
- **Backend Layer**: Express.js application
- **Data Layer**: MongoDB database
- **External Services**: Twilio SMS API

### Communication Flow
1. Frontend sends HTTP requests to backend
2. Backend processes requests and interacts with database
3. On appointment changes, SMS notifications are triggered
4. Twilio API handles SMS delivery
```

### For Database Documentation
```markdown
## Database Schema

[Include Entity Relationship Diagram]

### Tables
1. **USER** - Stores user/patient information
   - Role: Can be "patient" or "doctor"
   - PK: _id

2. **DOCTOR** - Stores doctor-specific information
   - Specialization: Medical field
   - Schedule: Map of available slots
   - PK: _id

3. **APPOINTMENT** - Booking records
   - Status: Pending → Confirmed → Completed/Cancelled
   - Links to both USER and DOCTOR

4. **UNAVAILABILITY** - Doctor availability exceptions
```

## Best Practices

### ✅ DO:
- Keep diagrams simple and readable
- Update diagrams when code changes significantly
- Use consistent colors and styling
- Include legends or key information
- Version control diagram files
- Document assumptions and constraints

### ❌ DON'T:
- Make diagrams too complex or cluttered
- Use outdated diagrams
- Mix too many diagram types in one document
- Forget to update diagrams with code changes
- Use diagrams without explanatory text
- Store only as images (keep source code)

## Diagram Export Checklist

When preparing diagrams for submission/documentation:

- [ ] All diagrams exported as high-quality images
- [ ] Diagrams properly sized (not too small/large)
- [ ] Clear titles and labels on each diagram
- [ ] Descriptions provided for each diagram
- [ ] Proper file naming (e.g., class-diagram.png)
- [ ] Organized in appropriate folder
- [ ] Links working in final document
- [ ] Dark/light theme compatibility checked
